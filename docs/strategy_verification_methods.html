トレード戦略の検証手法ガイド

バックテスト結果を「信じていい数字」にするための5つの手法 — 基礎用語・目的・現アプリへの適用

このドキュメントの目的
バックテストで「PF 2.14」「勝率58%」といった数字が出ても、それが実運用で再現する保証はない。
検証手法とは、バックテスト結果を疑い、どこまで信用できるかを定量的に測る技術のこと。
ここでは5つの手法を、何をしているのか → なぜ必要か → 現アプリでやるとどうなるか、の順で説明する。

目次

0.基礎用語 1.MAE / MFE 分析 — SLと利確の妥当性検証 2.Exit Timing 分析 — 保有期間の妥当性検証 3.条件別PnL分解 — エッジが効く条件の特定 4.ギャップリスク分析 — SLが機能しない場面の定量化 5.期間安定性検証 — 結果が偏っていないかの確認 6.優先度まとめ 7.検証(1) → 検知(3) の全体フロー

用語 0. 基礎用語

MAE Maximum Adverse Excursion
エントリーしてからexit（決済）するまでの間に、価格が最も不利な方向に動いた幅。
ロングなら「保有中の最安値 − エントリー価格」。
具体例
1,000円で買い → 保有中に950円まで下落 → 最終的に1,050円で決済
MAE = -5.0%（950円まで逆行した）
結果 = +5.0%（勝ちトレード）
→ 勝ったが、途中で-5%の含み損に耐える必要があった
MFE Maximum Favorable Excursion
エントリーしてからexitするまでの間に、価格が最も有利な方向に動いた幅。
ロングなら「保有中の最高値 − エントリー価格」。
具体例
1,000円で買い → 保有中に1,080円まで上昇 → 最終的に1,020円で決済
MFE = +8.0%（1,080円まで順行した）
結果 = +2.0%（+2%で終了）
→ +8%まで行ったのに+2%しか取れなかった。6%分の利益を逃した
PF Profit Factor
総利益 ÷ 総損失（絶対値）。
1.0以上なら利益が損失を上回っている。2.0以上は優秀とされる。
具体例
勝ちトレード合計: +500万円 / 負けトレード合計: -250万円
PF = 500 ÷ 250 = 2.0
WFO Walk-Forward Optimization
バックテスト期間を「学習期間」と「検証期間」に分けて時系列順にスライドしながら繰り返す手法。
過学習を検出するための標準的な方法。
具体例
14ヶ月のデータを4回に分けてテスト:
① 1-9月で学習 → 10-12月で検証
② 4-12月で学習 → 1-3月で検証
③ 7-3月で学習 → 4-6月で検証 ...
→ 検証期間が毎回プラスなら頑健。1回でもマイナスなら要注意。
OOS Out-of-Sample
戦略の設計・最適化に使わなかった期間のデータ。
「カンニングしていないテスト」に相当する。OOSでも利益が出れば戦略の信頼性が高い。
レジーム Market Regime
市場の「状態」のこと。上昇トレンド / 下降トレンド / レンジ / 高ボラ / 低ボラなど。
戦略はレジームによって効いたり効かなかったりする。
ギャップ Gap
前日終値と翌日始値の間の価格差。夜間に発生したニュース・先物変動が原因。
ストップロスはギャップを飛び越える。SL -3%を設定しても、ギャップダウン-5%で始まれば-5%で約定する。
2026/03/04 トレンドマイクロの実例
エントリー: 5,230円 / SL設定: 5,073円（-3%）
PTS（夜間取引）: 5,026円（-3.9%）
→ SL -3%を設定していたが、ギャップダウンにより-3.9%で約定する見込み。
SLは「最大損失の保証」ではなく「通常時の損切りライン」に過ぎない。

手法1 MAE / MFE 分析 — SLと利確の妥当性検証

Q. 何をしているのか？
全トレードについて、保有中の最大逆行（MAE）と最大順行（MFE）を記録する。
バックテスト結果は「エントリー価格」と「exit価格」しか記録しないことが多いが、 MAE/MFE は途中経過を記録する。

これにより「勝ちトレードの途中でどれだけ含み損が出ていたか」「負けトレードの途中でどこまで含み益があったか」がわかる。
Q. なぜする必要があるのか？
SL（ストップロス）の幅が適切かどうかを判断するため。

SL -3%が適切かどうかは「感覚」では判断できない。MAE分析により以下がわかる：

パターンA: SLが狭すぎる場合
→ 勝ちトレードの多くがMAE -2.5%〜-3.5%を記録している
→ つまり「一度-3%まで逆行してSLに刈られたが、その後反発して+5%まで行った」トレードが多い
→ SLを-4%に広げれば、これらのトレードを救える可能性がある

パターンB: SLが適切な場合
→ 勝ちトレードのMAEは-1%〜-2%に集中、負けトレードのMAEは-3%以深に集中
→ 勝ちと負けがSLラインで綺麗に分離されている
→ SL -3%は妥当

MFE分析は「利確タイミング」に使う：
→ MFEが+6%に到達するトレードが多いのに、7日引けで+2%になっている
→ 途中で利確すれば成績が改善する可能性がある

現アプリで実行すると何がわかるか
対象: Granville IFD戦略（SL -3%, 最大7営業日保有）

現在のアーカイブ（granville_ifd_archive.parquet）は14カラムで、entry_price と exit_price しか記録していない。 保有期間中の日足High/LowからMAE/MFEを計算して追加すれば：

・SL -3%で刈られたトレード（exit_type=SL）のうち、その後MFEが+3%以上に到達したケースの割合 → SLが狭すぎるかの判定
・勝ちトレードのMAE分布 → 勝ちの大半がMAE -1%以内なら、SLを-2%に狭めても成績が落ちない可能性
・MFEと最終リターンの差 → 途中で利確すべきだった利益の逸失額

必要な追加データ: なし（既存の日足parquetにHigh/Lowがある）
追加カラム: mae_pct, mfe_pct, mae_day, mfe_day

手法2 Exit Timing 分析 — 保有期間の妥当性検証

Q. 何をしているのか？
各トレードが何日目にexitしたかと、何日目に利益のピーク（MFE）に到達したかを記録する。

保有期間の設定が「7営業日」であるとき、実際の利益のピークがDay 2-3に集中しているなら、 7日間持つ意味がない。むしろDay 4以降は利益が削られている可能性がある。
Q. なぜする必要があるのか？
最適な保有日数を特定するため。

現在の「7営業日引け」は固定ルール。だが実際にはトレードごとに値動きのパターンが異なる：

即効型: Day 1-2で+4%まで行き、以後は横ばいか下落 → 早めに利確すべき
遅効型: Day 1-4は横ばい、Day 5-7で急伸 → 7日保有が正解
即死型: Day 1で-3% SLに到達 → 保有期間の問題ではなくエントリーの問題

これらの比率がわかれば、最適な保有日数や「途中利確ルール」の導入可否が判断できる。

日別リターン推移の見方

平均含み損益（%） +3% | * | * * +2% | * * ← Day 3がピーク。7日引けでは利益が減少 | * * +1% | * ← 最終exit（7日引け） | 0% *------|------|------|------|------|------|------ Entry Day1 Day2 Day3 Day4 Day5 Day6 Day7 もしこの形なら → 7日保有は長すぎる。Day 3-4での利確ルール追加を検討すべき
現アプリで実行すると何がわかるか
対象: Granville IFD戦略

現在 exit_type（SL / SMA20_touch / dead_cross / time_cut）は記録しているが、何日目にexitしたか（exit_day）は未記録。

追加すべき分析：
・exit_day の分布 → Day 1で即SLが多いなら、エントリーフィルターの問題
・MFE到達日の分布 → 利益ピークが何日目に来るか → 保有期間の最適化根拠
・Day別の平均リターン曲線 → 上図のような形で「何日目から利益が削られ始めるか」を可視化

追加カラム: exit_day（1-7）, mfe_day（1-7）

手法3 条件別PnL分解 — エッジが効く条件の特定

Q. 何をしているのか？
全トレードを条件ごとにグループ分けして、各グループのPnL・勝率・PFを比較する。

「全体で勝っている」だけでは不十分。どの条件で勝っていて、どの条件で負けているかを分離する。
負けている条件を除外すれば、戦略全体の成績が改善する。
Q. なぜする必要があるのか？
戦略のエッジ（優位性）がどの条件に依存しているかを特定するため。

バックテストで「PF 2.14、14ヶ月で+418万」と出ても、内訳を見ると：

・uptrend時: PF 3.2, +380万（利益の91%がここ）
・downtrend時: PF 1.1, +38万（ほぼトントン）

この場合、「この戦略はuptrend時に強い」が正確な結論。downtrend時に使うかどうかは別途判断が必要。

分解軸の例：
・市場レジーム: uptrend / downtrend（N225 vs SMA20）
・シグナル種別: Tier 1（B4） / Tier 2（sigA deep）/ Tier 3（sigA mod）/ Tier 4（sigB）
・セクター: 業種ごとの勝率・PF
・先物変動: 前日先物変動 >3% / <3%（Grokが機能しない閾値の検証）
・曜日: 月〜金で偏りがあるか
・ボラティリティ: VI水準ごとの成績差

現アプリで実行すると何がわかるか
対象: Granville IFD / Grok 両方

Granville IFD: アーカイブに market_uptrend, ci_expand, signal_type, sector が既にある。 これらを軸に分解するだけで「どのTierが本当に稼いでいるか」「downtrend時は止めるべきか」が判明する。

Grok: アーカイブに futures_change_pct, is_extreme_market が既にある。 「先物変動 >3%の日」だけ抽出してPnLを見れば、戦争ラリーや選挙相場でGrokが機能しない閾値を定量化できる。

追加データ: 不要（既存カラムで分解可能）

手法4 ギャップリスク分析 — SLが機能しない場面の定量化

Q. 何をしているのか？
前日引け→翌日寄りの価格差（ギャップ）を全トレードで記録し、 SLが想定通りに機能したかどうかを検証する。

SL -3%を設定していても、ギャップダウンで-5%から始まれば-5%で約定する。 この「SLの滑り（スリッページ）」がどれだけ発生しているかを定量化する。
Q. なぜする必要があるのか？
バックテストの「SL -3%」が実運用でも-3%で約定する保証はないから。

バックテストでは「始値がSL以下なら始値で約定」とモデル化されることが多い。 これは正しいが、「SLに引っかかるケースのうち何割がギャップ由来か」は別途確認が必要。

もしSL発動の40%がギャップ由来で、平均スリッページが-1.5%だとすると：
→ バックテスト上の「SL -3%」は実質「SL -4.5%」で約定している
→ バックテストの損失額は現実を正しく反映していない可能性

逆にギャップが小さい銘柄（大型株、流動性高い）だけに絞れば、SLの信頼性が上がる。

ギャップリスクの構造

前日引け 翌日寄り ケース1: 通常のSL発動（ギャップなし） ────●SL(-3%)──────●寄り(-2.5%)──→ SL価格で約定。想定通り。 ↑ SL設定 ↑ 寄りがSL以上なのでSL到達時点で約定 ケース2: ギャップダウンでSL突破 ────●SL(-3%)──✕──────●寄り(-5%)──→ 寄り値で約定。-5%の損失 ↑ SL設定 ↑ SLを飛び越え ↑ ここで初めて約定可能 ケース3: ストップ安（流動性消失） ────●SL(-3%)──✕──────✕──────●翌々日(-12%)──→ 約定すらできない ↑ SL設定 ↑ ストップ安で値がつかない ↑ やっと約定
現アプリで実行すると何がわかるか
対象: Granville IFD戦略

現在 exit_type=SL のトレードは記録されているが、SL価格と実際のexit価格の差（スリッページ）は未記録。

追加すべき分析：
・SLトレードのスリッページ分布 → SL発動時、平均何%余計に損しているか
・保有期間中の最大ギャップダウン → 夜間リスクの大きさ
・ギャップサイズ vs 銘柄の時価総額・流動性 → 大型株に絞ればギャップリスクは減るか

現在の「株価2万円未満」フィルターは流動性と無関係。出来高や時価総額でのフィルター追加の根拠データになる。

追加カラム: overnight_gap_pct, sl_slippage_pct
注意: ギャップリスクはバックテストでは過小評価されやすい。 バックテストは「始値で約定」を前提とするが、実際には寄り付き直後の板が薄く、 成行注文が想定より不利な価格で約定することがある。特に小型株・低出来高銘柄で顕著。

手法5 期間安定性検証 — 結果が偏っていないかの確認

Q. 何をしているのか？
バックテスト全体の結果を四半期・月単位に分割して、 各期間のPnL・勝率・PFを比較する。

全期間で均等に利益が出ているのか、特定の1-2ヶ月に利益が集中しているのかを確認する。
Q. なぜする必要があるのか？
「たまたま良い期間に助けられただけ」でないかを確認するため。

例えば14ヶ月で+418万のうち：

パターンA（安定型）: 12ヶ月中10ヶ月がプラス、最大マイナス月が-30万 → 頑健
パターンB（偏り型）: 2025年8月の急落相場で+350万、他の月は合計+68万 → 脆弱

パターンBの場合、あの急落がなければPF 1.2程度の凡庸な戦略。 将来同じような急落が来る保証はないので、この戦略を「強い」と判断するのは危険。

分割の粒度：
・月別: 最低限。14ヶ月なら14データポイント
・四半期別: トレンドの変化を大きく捉える
・年別: 12年の長期データがあるGranville 8法則分析では可能

判定基準: 月単位でプラスの割合が70%以上なら安定的。50%以下なら特定月への依存度が高い。

現アプリで実行すると何がわかるか
対象: Granville IFD / Grok 両方

Granville IFD: サーバーAPIのsummaryに月別統計は既にある。 ただし「月単位のプラス率」「最大マイナス月」「上位3ヶ月で利益の何%を占めるか」は未計算。 これを追加すれば安定性を一目で判断できる。

Grok: backtest/summaryに日別統計はあるが、月別のPF・勝率は未集計。 4ヶ月分（2025/11〜2026/02）と期間が短いため、月別より週別が適切。

重要な検証ポイント: Granville IFDの+418万のうち、uptrend月とdowntrend月の内訳。 もし利益の90%がuptrend月に集中していれば、downtrend時は戦略を停止すべき根拠になる。

追加データ: 不要（既存アーカイブで計算可能）

構造 7. 検証(1) → 検知(3) の全体フロー

現アプリでの実装イメージ
Phase 1（検証）: バックテストスクリプトにMAE/MFE等のカラム追加 → アーカイブparquet更新
Phase 2（可視化）: サーバーAPIとフロントエンドに分析ビュー追加
Phase 3（検知）: 日次パイプラインに閾値チェック追加 → フロントエンドにアラート表示

Phase 1-2で検証結果を見てから、Phase 3の閾値を人間が決める。
Phase 3の閾値をAIに決めさせない。それが記事の言うHuman-in-the-Loopの核心。
