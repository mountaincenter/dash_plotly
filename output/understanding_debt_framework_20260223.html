1. 飛行機の比喩 - 何が理解負債で、何がそうでないか

飛行機がなぜ飛ぶかを理解しなくても、安全に目的地に到着できる。 LLM を用いた開発でも同じ原理が成り立つ。

レイヤー飛行機あなたのシステム理解負債？
物理法則揚力の方程式Python のメモリ管理、TCP/IP No
機体構造エンジンの仕組みDocker、AWS App Runner No
操縦系統自動操縦の制御ロジックDash/React のレンダリング、CSS No
計器の読み方高度計、燃料計API のレスポンス形状、エラーコード 境界
目的地の選択どこに行くかどの戦略でどの銘柄を分析するか 必須
天候判断飛ぶか飛ばないかデータの信頼性、バックテストの限界 必須

判断基準は1つ

「それを理解していないと、間違った意思決定をするか？」

  • No → 理解負債ではない。委任して良い
  • Yes → 理解負債。解消が必要

3. LLM 時代に価値が上がる能力

実行力が無限に近づくほど、何を解くか（イシューの質）が成果の 100% を決める。
間違ったイシュー + 完璧な実装 = 無価値
正しいイシュー + LLM の実装 = 価値

5. 書籍以外 - 最も効果的なトレーニング

判断ログを取る

一番効くのは「判断 → 結果 → 振り返り」のログを記録すること。

判断 → 根拠 → 結果 → 振り返り

例: 「この戦略が有効だと判断した → 根拠は X → 結果は Y → 判断は正しかった/間違っていた、なぜなら〜」

イシューの質・不確実性判断・システム思考の全てを同時に鍛える。
書籍は地図、ログは実走。
